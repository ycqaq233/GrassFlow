| **工具系统** | 无 | 无法让 AI 调用工具 |

## 二、目标架构

### 愿景

让 GrassFlow 成为一个**交互式多 Agent 编排平台**，用户可以：
1. 像 Claude Code 一样与 AI 对话
2. 在对话中创建工作流、编辑 Agent
3. 让 AI 自动创建和调用工作流
4. 通过 Skills 和 MCP 扩展能力

### 架构图

```
GrassFlow 目标架构:

┌─────────────────────────────────────────────────────────────┐
│                    REPL 会话层                               │
│  ┌─────────────────────────────────────────────────────────┐│
│  │  Session Manager (SQLite 持久化)                        ││
│  │  Context Compressor (token 检测 + 摘要)                 ││
│  │  Input Router (状态机: 空闲/运行中/审批/中断)            ││
│  └─────────────────────────────────────────────────────────┘│
├─────────────────────────────────────────────────────────────┤
│                    AI 对话引擎                               │
│  ┌─────────────────────────────────────────────────────────┐│
│  │  LLM Protocol Layer (四维抽象)                          ││
│  │  Tool Executor (并发安全 + 权限控制)                    ││
│  │  Error Classifier (结构化错误)                          ││
│  │  Circuit Breaker (熔断器)                               ││
│  └─────────────────────────────────────────────────────────┘│
├─────────────────────────────────────────────────────────────┤

│  ┌─────────────────────────────────────────────────────────┐│

│  │  MCP Client/Server (双向集成)                           ││
```

## 三、实施计划

### 阶段 1: 基础架构 (1-2周)

**目标**: 建立可扩展的 LLM 和工具基础

| 任务 | 说明 | 优先级 |
|------|------|--------|
| LLM 协议抽象层 | 参考 OpenCode 四维模型 | 🔴 高 |
| 工具注册表 | 参考 Hermes 自注册模式 | 🔴 高 |
| Skills 系统 | YAML + Markdown 格式 | 🔴 高 |
| 错误分类器 | 结构化错误枚举 | 🔴 高 |

**交付物**:
- `core/llm_protocol.py` - LLM 协议抽象
- `core/tool_registry.py` - 工具注册表
- `core/skills.py` - Skills 管理器
- `core/error_classifier.py` - 错误分类器

### 阶段 2: 会话系统 (2-3周)

**目标**: 实现交互式 REPL 会话

| 任务 | 说明 | 优先级 |
|------|------|--------|
| REPL 会话管理 | 持续对话能力 | 🔴 高 |
| 上下文压缩 | token 超限检测 + 摘要 | 🟡 中 |
| 会话持久化 | SQLite 存储 | 🟡 中 |
| 断点恢复 | 从断点继续执行 | 🟡 中 |

**交付物**:
- `tui/session.py` - 会话管理器
- `tui/context_compressor.py` - 上下文压缩
- `tui/repl.py` - REPL 主循环
- 数据库表扩展

### 阶段 3: 工具系统 (2-3周)

**目标**: 实现工具调用和 MCP 集成

| 任务 | 说明 | 优先级 |
|------|------|--------|
| 内置工具 | shell, read, write, glob, grep | 🔴 高 |
| MCP Client | 连接外部 MCP 服务器 | 🟡 中 |
| MCP Server | 暴露工作流为工具 | 🟡 中 |
| 权限控制 | allow/deny/ask 三级权限 | 🟡 中 |

**交付物**:
- `tools/` 目录 - 内置工具
- `core/mcp_client.py` - MCP 客户端
- `core/mcp_server.py` - MCP 服务器
- `core/permission.py` - 权限控制

### 阶段 4: 高级特性 (2-3周)

**目标**: 完善用户体验和系统健壮性

| 任务 | 说明 | 优先级 |
|------|------|--------|
| 熔断器 | 连续失败触发，冷却后半开 | 🟡 中 |
| Doom Loop 检测 | 防止 Agent 陷入死循环 | 🟡 中 |
| 渐进式披露 | 最小信息 → 完整内容 | 🟢 低 |
| AI 自动创建工作流 | 让 AI 生成 DSL | 🟢 低 |

**交付物**:
- `core/circuit_breaker.py` - 熔断器
- `core/doom_loop.py` - Doom Loop 检测
- Skills 渐进式披露
- AI 工作流生成器

## 四、技术决策

### 4.1 保留的设计

| 设计 | 理由 |
|------|------|
| DAG 拓扑排序 | 核心差异化，多 Agent 编排的基础 |
| 声明式 DSL | 降低门槛，AI 可生成 |
| Rich TUI | Python 生态成熟，轻量级 |
| Textual 编辑器 | 交互式编辑已实现 |
| 多级配置 | 灵活的配置管理 |

### 4.2 需要重构的设计

| 设计 | 现状 | 目标 |
|------|------|------|
| CLI 入口 | 单次命令执行 | REPL 会话模式 |
| LLM 调用 | 直接调用 LiteLLM | 协议抽象层 |
| 工具系统 | 无 | 统一注册表 |
| 错误处理 | 字符串匹配 | 结构化分类 |

### 4.3 新增的设计

| 设计 | 说明 |
|------|------|
| 会话管理 | SQLite 持久化 + 断点恢复 |
| 上下文压缩 | token 检测 + 摘要 Agent |
| Skills 系统 | YAML + Markdown 格式 |
| MCP 集成 | Client + Server 双向 |
| 权限控制 | allow/deny/ask 三级 |

## 五、风险和缓解

| 风险 | 影响 | 缓解措施 |
|------|------|---------|
| 重构范围过大 | 现有功能可能被破坏 | 保持向后兼容，逐步迁移 |
| LLM 协议复杂 | 实现周期长 | 先支持 OpenAI，再扩展 |
| MCP 协议变化 | 需要持续跟进 | 抽象接口，隔离变化 |
| 性能问题 | 多 Agent 并行可能慢 | 异步优化，资源限制 |
