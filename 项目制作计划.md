## 当前进度总览

### 已实现的核心能力

1. **声明式 DSL v2** -- Component + Workflow + Connection + Port 完整类型系统（v2.1 规范）
2. **DAG 引擎** -- 拓扑排序、环检测、并行分组、port-aware 数据流
3. **工作流执行** -- CLI + REPL 端到端：解析 .gf → 创建 Agent → Scheduler 执行 → 结果展示
4. **Stream 模式** -- 流式处理：逐项触发 + shared 累积 / independent 隔离
5. **工具调用** -- LLMAgent 支持工具调用迭代循环（最多 10 轮）
6. **Per-agent 权限过滤** -- Component permission 声明 → create_filtered_registry → 每个 agent 独立工具集
7. **Scheduler 事件回调** -- 10 种事件类型（AGENT_START/COMPLETE/FAIL 等），驱动监控面板
8. **REPL 智能编排** -- 意图检测（4 种模式）→ 自动生成 DSL → /run 执行
9. **AI 工作流生成** -- /generate 命令：自然语言 → DSL → 验证 → 保存
10. **组件系统** -- ComponentRegistry 发现/注册/查询/搜索/导入导出
11. **交互式 REPL + 流式输出** -- hermes 模式 patch_stdout + 终端原生滚动
12. **多 Provider 支持** -- DeepSeek/OpenAI/Anthropic/Ollama，opencode 风格配置
13. **工具系统** -- shell/read/write/glob/grep/webfetch 内置工具 + MCP 集成 + 统一注册表
14. **安全机制** -- 权限控制(allow/deny/ask) + 熔断器 + Doom Loop 检测
15. **Agent Loop** -- 完整的 Agent 主循环，支持工具调用和流式输出
16. **会话管理** -- SQLite 持久化 + 断点恢复 + 上下文压缩
17. **监控面板** -- 回调驱动 + 连接可视化 + 双模式（全屏/嵌入）
18. **Docker 容器化** -- Dockerfile + docker-compose.yml，一条命令运行

## 工作流驱动开发经验

### 有效的编排模式

1. **多 agent 并行扫描 + 对比 + 修复**（47 agents，101 发现，100 修复）
   - 每个 agent 专注 1-2 个文件，避免上下文溢出
   - 先扫描 → 对比 hermes 参考 → 精确修复 → 验证
   - 关键：agent 上下文只有 256k，必须拆分任务

2. **研究 + 设计 + 重写 + 验证**（8 agents，TUI hermes 模式重构）
   - 4 个研究 agent 并行研究 hermes 的 REPL/滚动/渲染/布局
   - 综合设计 → 2 个 agent 并行重写 layout.py + repl.py
   - 验证 agent 检查一致性

### 失败的模式

1. **单 agent 做所有事情** -- 上下文溢出，修复不完整
2. **不理解参考架构就改代码** -- 鼠标滚轮修了 6 次才解决，根因是架构方向错误
3. **工作流验证不充分** -- agent 报告"验证通过"但实际有遗漏

### 关键教训

- **模型能力不足可以用完备的工作流弥补**，但工作流必须基于对参考代码的深入理解
- **先理解架构再改代码**，不要在错误的方向上反复修补
- **每个 agent 的任务必须足够小**，256k 上下文是硬限制
- **验证必须包含实际运行测试**，不能只靠代码审查
