## 工作流驱动开发经验

### 有效的编排模式

1. **多 agent 并行扫描 + 对比 + 修复**（47 agents，101 发现，100 修复）
   - 每个 agent 专注 1-2 个文件，避免上下文溢出
   - 先扫描 → 对比 hermes 参考 → 精确修复 → 验证
   - 关键：agent 上下文只有 256k，必须拆分任务

2. **研究 + 设计 + 重写 + 验证**（8 agents，TUI hermes 模式重构）
   - 4 个研究 agent 并行研究 hermes 的 REPL/滚动/渲染/布局
   - 综合设计 → 2 个 agent 并行重写 layout.py + repl.py
   - 验证 agent 检查一致性

### 失败的模式

1. **单 agent 做所有事情** -- 上下文溢出，修复不完整
2. **不理解参考架构就改代码** -- 鼠标滚轮修了 6 次才解决，根因是架构方向错误
3. **工作流验证不充分** -- agent 报告"验证通过"但实际有遗漏

### 关键教训

- **模型能力不足可以用完备的工作流弥补**，但工作流必须基于对参考代码的深入理解
- **先理解架构再改代码**，不要在错误的方向上反复修补
- **每个 agent 的任务必须足够小**，256k 上下文是硬限制
- **验证必须包含实际运行测试**，不能只靠代码审查
